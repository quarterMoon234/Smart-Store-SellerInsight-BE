# 프로젝트 방향 전환 문서

## 1. 문서 목적

이 문서는 본 프로젝트가 어떤 초기 기획으로 시작되었고, 개발 과정에서 어떤 현실적인 제약을 확인했으며, 그 결과 어떤 기준으로 방향을 재설계했는지를 설명하기 위해 작성했다.

면접에서는 단순히 "기획이 바뀌었다"는 사실보다,

- 어떤 가정을 했는지
- 어떤 근거로 가정이 깨졌는지
- 그 상황에서 무엇을 유지하고 무엇을 버렸는지
- 최종적으로 어떤 기술 역량을 보여주기로 했는지

를 일관되게 설명하는 것이 중요하다고 판단했다.

---

## 2. 초기 기획

초기 기획은 다음과 같았다.

> 네이버 스마트스토어/커머스 API를 활용해 판매자 데이터를 수집하고, 주문 감소, 상품 편중, 유입 변화 같은 신호를 분석한 뒤 판매자의 다음 행동을 제안하는 AI 피드백 서비스를 만든다.

초기 문제 정의 자체는 명확했다.

- 네이버 판매자 관리 페이지는 주문, 상품, 리뷰, 유입 관련 정보를 제공하지만 화면이 분산되어 있다.
- 판매자는 데이터를 "조회"할 수는 있지만, 데이터를 기반으로 "다음 행동"을 바로 결정하기 어렵다.
- 따라서 파편화된 데이터를 수집하고 정규화한 뒤, 핵심 지표와 이상 징후를 묶어 행동 가능한 피드백으로 재구성하는 서비스는 충분히 의미가 있다고 판단했다.

초기에는 다음 요소를 핵심 가치로 봤다.

- 외부 API 연동
- 데이터 수집 및 정규화
- 지표 집계
- 규칙 기반 인사이트
- 선택적 AI 요약

즉, 처음부터 AI 자체보다 백엔드 분석 시스템으로 보는 관점이 강했다.

---

## 3. 실제 조사 후 확인한 현실 제약

프로젝트 진행 중 네이버 커머스 API 공식 문서를 기준으로 인증 구조와 솔루션 등록 절차를 다시 확인했고, 이 과정에서 중요한 제약을 발견했다.

### 3-1. 실제 판매자 데이터 연동은 단순 공개 API 호출이 아니었다

판매자 데이터에 접근하려면 단순한 공개 API 호출이 아니라 다음 조건이 필요했다.

- 네이버 커머스API센터 솔루션 등록
- 애플리케이션 ID / 애플리케이션 Secret 발급
- 솔루션 구독 연동
- JWT/JWE 기반 판매자 식별 정보 수신
- `accountUid` 기반 `SELLER` 토큰 발급

즉 "개인 프로젝트에서 바로 clientId/clientSecret 하나 받아 붙이는 수준"이 아니었다.

### 3-2. 개인 자격으로는 실연동이 사실상 어려웠다

공식 문서 확인 결과, 개발사 입점과 솔루션 등록은 현재 국내 법인 사업자 계정 중심으로 운영되고 있었다.

이 의미는 명확하다.

- 개인 자격으로는 솔루션 등록 자체가 막힐 수 있다.
- 솔루션 등록이 안 되면 실제 애플리케이션 ID/Secret을 발급받기 어렵다.
- JWT 검증용 Public Key 및 판매자 `accountUid` 확보도 어렵다.
- 결과적으로 실제 판매자 데이터를 받아오는 end-to-end 실연동 검증은 불가능하거나 매우 제한적이다.

### 3-3. 일부 분석 API는 추가 제약이 있었다

고객 데이터, 마케팅 분석, 판매 분석, 실시간 분석 계열은 브랜드스토어 전용이거나 API 데이터 솔루션 구독이 필요한 범위가 섞여 있었다.

즉 초기 기획처럼 "범용 판매자 대상 고급 분석 서비스"를 바로 구현하는 것은 현실적으로 범위가 맞지 않았다.

## 4. 여기서 내린 판단

이 시점에서 선택지는 두 가지였다.

### 선택지 A. 프로젝트 중단

실제 네이버 판매자 데이터를 받을 수 없다면 프로젝트 목적이 사라졌다고 보고 중단하는 방식이다.

### 선택지 B. 프로젝트 리포지셔닝

네이버 실연동 SaaS가 아니라, 외부 커머스 데이터를 입력받아 판매자 인사이트를 생성하는 백엔드 시스템으로 방향을 바꾸는 방식이다.

나는 **선택지 B를 택했다.**

이유는 내 프로젝트의 1차 목적이 "실서비스 출시"보다 "백엔드 포트폴리오"에 있었기 때문이다.

내가 이 프로젝트를 통해 보여주고 싶었던 역량은 원래도 다음에 가까웠다.

- 외부 플랫폼 연동을 고려한 설계
- 데이터 모델링
- 정규화
- 배치 집계
- 규칙 엔진
- 운영 모니터링

이 역량들은 실연동이 막혀도 여전히 충분히 구현하고 증명할 수 있다고 판단했다.

## 5. 변경 전/후 프로젝트 정의

### 변경 전

`네이버 커머스 API 기반 판매자 AI 피드백 서비스`

### 변경 후

`판매자 데이터를 수집·정규화·분석해 다음 행동을 추천하는 커머스 인사이트 백엔드 시스템`

이 변경의 핵심은 다음과 같다.

- 네이버는 더 이상 서비스 정체성이 아니라 입력 어댑터 후보 중 하나다.
- 실제 핵심은 판매자 데이터 처리 파이프라인과 인사이트 생성 엔진이다.
- AI는 선택 요소이고, 주인공은 백엔드 분석 로직이다.

## 6. 무엇을 유지했고, 무엇을 바꿨는가

### 유지한 것

- 판매자 관점의 문제 정의
- 행동 가능한 인사이트 제공이라는 목표
- 규칙 기반 분석 우선 구조
- 운영성과 관측성 중시

### 축소하거나 보류한 것

- 네이버 실판매자 계정 연동
- `SELLER` 토큰 실발급 검증
- 솔루션 구독/JWT/accountUid 실환경 처리
- 브랜드스토어 전용 고급 분석 API 의존 기능

### 새로 강화한 것

- CSV Import 기반 데이터 수집
- Sample Data / Mock Adapter 기반 입력 계층
- 입력 어댑터와 분석 코어의 분리
- 외부 플랫폼 제약을 고려한 구조적 확장성

## 7. 최종 개발 방향

최종적으로 이 프로젝트는 아래 구조를 목표로 한다.

### 7-1. 입력 계층

- CSV 업로드
- 샘플 데이터 시드
- Mock 외부 API 어댑터

### 7-2. 저장 및 정규화 계층

- `seller`
- `product`
- `order`
- `order_item`
- `import_job`
- `daily_metric`
- `insight`
- `recommendation`

### 7-3. 분석 계층

- 최근 7일 주문 증감
- 환불/취소 비율 변화
- 상품 편중도
- 장기 미판매 상품 증가
- 품절 리스크

### 7-4. 인사이트 계층

- 규칙 기반 이상 징후 감지
- 우선순위가 있는 액션 추천
- 필요 시 템플릿 또는 LLM 기반 문장화

### 7-5. 운영 계층

- 배치/동기화 상태 추적
- Prometheus 메트릭
- Loki 로그 수집
- Grafana 대시보드

## 8. 이 방향이 백엔드 포트폴리오로 여전히 가치 있는 이유

이 프로젝트는 실연동 제약이 생긴 뒤에도 다음 역량을 보여줄 수 있다.

### 8-1. 도메인 모델링

원본 입력을 그대로 쓰지 않고, 판매자/주문/상품/지표/인사이트라는 내부 모델로 정규화한다.

### 8-2. 데이터 파이프라인 설계

파일 입력, 검증, 정규화, 적재, 집계, 인사이트 생성까지 이어지는 흐름을 구현할 수 있다.

### 8-3. 배치 및 운영성

집계 배치, 실패 추적, 재실행 가능성, 메트릭 노출, 로그 수집까지 포함해 운영 가능한 구조를 보여줄 수 있다.

### 8-4. 규칙 기반 판단 로직

AI에 의존하지 않고 백엔드에서 직접 비즈니스 신호를 계산하고 판단하는 구조를 구현할 수 있다.

### 8-5. 외부 제약 대응 능력

플랫폼 정책이나 권한 제한이 생겼을 때 프로젝트를 버리는 대신, 목표를 다시 정의하고 구조를 재설계한 판단 과정을 설명할 수 있다.

개인적으로는 이 마지막 항목이 면접에서 특히 중요하다고 본다.

실무에서도 항상 "가장 이상적인 방식"으로만 개발할 수는 없고, 제약 안에서 목적을 재정의하는 능력이 중요하기 때문이다.

## 9. 면접에서 이렇게 설명할 계획

면접에서는 이 프로젝트를 다음과 같이 설명할 계획이다.

> 처음에는 네이버 커머스 API 기반 판매자 인사이트 서비스를 기획했습니다.  
> 하지만 개발 과정에서 솔루션 등록, 키 발급, JWT 기반 판매자 식별 등 실연동에 필요한 절차가 개인 자격으로는 제한적이라는 현실 제약을 확인했습니다.  
> 그래서 프로젝트를 중단하지 않고, 외부 커머스 연동을 고려한 구조는 유지하되 입력 계층을 CSV/Mock Adapter로 대체해 판매자 데이터 정규화, 집계, 규칙 기반 인사이트 생성, 운영 모니터링에 집중하는 백엔드 포트폴리오 프로젝트로 재설계했습니다.

이 설명의 목적은 단순 변명이 아니라,

- 현실 제약을 확인한 과정
- 그에 대한 기술적 판단
- 내가 끝까지 가져가기로 한 핵심 가치

를 분명하게 전달하는 데 있다.

---

## 10. 현재 기준 최종 한 줄 정의

> 외부 커머스 플랫폼 연동을 고려해 판매자 데이터를 정규화하고, 일별 지표와 규칙 기반 인사이트를 생성해 판매자의 다음 행동을 추천하는 커머스 인사이트 백엔드 시스템

---

## 11. 참고

본 방향 전환 판단은 네이버 커머스API센터 공식 문서 확인을 바탕으로 했다.

- 개발사 입점 동의
- 마켓 입점 프로세스
- 솔루션 등록 개발사 정보 입력
- 인증 / SELLER 토큰 발급 구조

실제 실연동 여부는 개인 계정 조건, 법인 자격, 솔루션 등록 가능 상태에 따라 달라질 수 있으므로, 본 프로젝트는 실연동 자체보다 백엔드 설계 역량을 보여주는 방향으로 범위를 조정했다.
